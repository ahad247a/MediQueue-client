# 🏥 MediQueue – Tutor Booking System

MediQueue is a premium, full-stack Tutor Booking and Management System tailored for specialized academic fields (such as Medical, Engineering, or Professional streams). Built using **Next.js (App Router)** and **Express.js**, the platform features an advanced queue-based or slot-based booking workflow. It allows students to seamlessly discover available tutor slots, while empowering tutors to post, update, track, and manage their tuition schedules effortlessly.

## ✨ Features

- **🔐 Private Access Control:** Integrated robust authentication via **Better-Auth** supporting secure Email/Password credentials and Google Social sign-ins to protect private dashboards.
- **📊 Interactive Slots Matrix ("My Tutors"):** Dedicated control room layout for tutors to dynamically manage remaining slots, hourly fees, and availability.
- **⚡ Optimistic Core Actions:** Instant UI rendering on `PATCH` (Update) and `DELETE` requests utilizing React state synchronization, preventing laggy page refreshes.
- **⏳ Real-time Slot Indicators:** Visual badges that color-code and flag empty or full slots (`0 Slots` warning flags) instantly to manage incoming student flow.
- **🌓 Adaptive Theme Engine:** Continuous Light and Dark mode variations mapped smoothly across tables, inputs, and modals using native Tailwind directives.

## 🛑 Challenging Parts & Solutions

Building a specialized system like **MediQueue** introduced complex full-stack challenges. Below are the key engineering hurdles and how they were solved:

### 1. Asynchronous Queue/Slot Consistency & Data Integrity
- **Challenge:** HTML `<input type="number">` fields natively return values as a `String` from `e.target.value`. Posting these strings directly into MongoDB triggers validation breaks when executing queries like `$inc` or mathematical decrements for remaining booking slots.
- **Solution:** Enforced strict data cast layers immediately inside the React state lifecycle. Constructed an interim sanitized object (`cleanedData`) using explicit `Number()` constructors to format `hourlyFee` and `totalSlot` right before dispatching the `PATCH` or `POST` payload.

### 2. Preventing State Amnesia and UI Collapses in Modals
- **Challenge:** Triggering data updates inside the table via modal inputs often threatened to erase implicit parameters generated by MongoDB (such as `_id`, `tutorEmail`, or timestamps) if the state was simply overwritten by the raw form inputs.
- **Solution:** Refactored the array map iterator callback (`prevTutors.map(...)`) to perform a deliberate deep shallow-merge using object spreading: `{ ...t, ...cleanedData }`. This ensures backend-controlled values remain perfectly intact while surface values morph gracefully.

### 3. Dynamic Hydration Conflicts & Field Contrast in Dark Mode
- **Challenge:** While using complex components like Modals coupled with Tailwind's adaptive dark-mode config, default text attributes in browser form fields occasionally rendered user input completely invisible or illegible due to poor contrast.
- **Solution:** Configured strict, absolute background and foreground text controls directly within the modal input styles (`text-slate-800 dark:text-slate-100 bg-transparent border-slate-200 dark:border-slate-800`) guaranteeing high readability under any user system theme preference.

### 4. Client-side Session Protection vs. Flash Content (Flicker)
- **Challenge:** Guarding private pages from unauthorized users often created a brief "flash" of the form layout before the client routing system successfully kicked unauthenticated requests out to the `/login` route.
- **Solution:** Structured an absolute execution gate using the structural loading indicator wrapper attached directly to Better-Auth's active session state hook (`isPending`). The page strictly delays interface parsing until credentials resolve cleanly.
